# Debate Mode 설계 v2
# (Round 1 Codex 비판 4개 [BLOCK] 반영)

## 목적
기획/아키텍처 결정 전, Leader의 설계안을 Codex xhigh가 적대적으로 검토하여
blind spot을 제거하고 더 나은 결정을 도출한다.

## 트리거
- 사용자가 명시적으로 "debate", "아키텍처 토론" 요청 시 (일반 "검토해줘"는 제외)
- Leader 판단 기준:
  - 기술 선택지가 2개 이상이고 팀 전체에 영향을 미칠 때
  - 되돌리기 어려운(irreversible) 결정일 때
  - 불확실성/영향도/복잡도 점수가 임계치 이상일 때 (각 1-3점, 합 6+ → debate 진입)

## 참여자 & 역할 분리 ([BLOCK] #1 해결)
- **Proposer**: Leader (Sonnet+thinking) — 설계안 초안 작성
- **Critic**: Codex exec xhigh — 비판적 검토 (read-only, 코드 수정 권한 없음)
- **Judge**:
  - 저위험(점수 6-7): Leader가 판단, 근거 문서화
  - 고위험(점수 8-9): AskUserQuestion으로 사용자가 최종 판단
  - 연구 참고: Proposer=Judge 구조는 자기확증 편향 발생. 고위험 시 반드시 외부 Judge 필요.

## 분류 기준 정의 ([BLOCK] #2 해결)
- **[BLOCK]**: 현재 설계로는 핵심 요구를 충족할 수 없거나, 출시/구현이 불가능한 경우
  - 예: 데이터 정합성 파괴, 보안 취약점 내재, SLO 달성 불가
- **[TRADEOFF]**: 요구는 충족하나 비용/복잡도/유지보수 부담이 명확히 증가하는 경우
  - 예: 성능 개선 위해 복잡도 2배 증가, 빠른 출시 위해 기술 부채 의도적 발생
- **[ACCEPT]**: 즉시 반영 가능한 개선점. 수정하지 않으면 품질 저하가 예상됨
  - 예: 누락된 에러 핸들링, 엣지케이스 미처리

## Critic 출력 강제 포맷 ([BLOCK] #3 해결)
Codex에 요청할 때 반드시 아래 포맷으로 출력하도록 지시:

```
[BLOCK|TRADEOFF|ACCEPT] {분류}: {한 줄 요약}
- 문제: {구체적으로 무엇이 문제인가}
- 영향: {발생할 수 있는 결과}
- 근거: {어떤 사실/제약/원칙에 기반하는가 — 의견만이면 TRADEOFF 이하}
- 수정안: {구체적인 대안}
- 미반영 리스크: {이 비판을 무시하면 어떤 일이 생기는가}
```

증거 없이 의견만 있는 비판 → Proposer가 "근거 부족"으로 기각 가능.

## 설계안 입력 형식 ([BLOCK] #4 해결 — 비기능 요구 추가)
```
## 결정 대상
{무엇을 결정하는가}

## 컨텍스트
{프로젝트 현황, 기존 아키텍처}

## 제안 방향
{채택하려는 방향과 근거}

## 고려한 대안
{검토한 대안과 기각 이유}

## 비기능 요구 (필수)
- 성능/처리량: {예: 응답시간 < 200ms, TPS 1000+}
- 비용 한계: {예: 월 $X 이내}
- 보안/규제: {예: GDPR, 암호화 필요}
- 가용성/SLO: {예: 99.9% uptime}
- 롤백 기준: {언제 되돌릴 것인가}

## 우려사항 (자기검열)
{스스로 의심되는 부분}

## 위험도 점수 (합산 → Judge 결정)
- 불확실성: 1(낮음) / 2(보통) / 3(높음)
- 영향 범위: 1(로컬) / 2(도메인) / 3(전체 시스템)
- 복잡도: 1(단순) / 2(보통) / 3(복잡)
- 합계: {X}/9 → 6+ 시 debate 진입, 8+ 시 사용자 최종 판단
```

## 라운드 구조 (최대 2라운드)

### Round 1
1. Leader가 설계안 작성 (위 형식)
2. Codex에 전송:
   ```
   codex exec -c model_reasoning_effort=xhigh -s read-only -C {project-path} \
     "다음 설계안을 비판적으로 검토하라.
      각 비판은 반드시 아래 포맷으로 작성하라:
      [BLOCK|TRADEOFF|ACCEPT]: 한 줄 요약 / 문제 / 영향 / 근거 / 수정안 / 미반영 리스크
      BLOCK=핵심 요구 위반, TRADEOFF=비용/복잡도 증가, ACCEPT=즉시 개선 가능
      설계안: {design}"
   ```
3. Leader가 Codex 비판 분석:
   - [BLOCK] 없음 → 조기 종료 (TRADEOFF/ACCEPT만 문서화 후 결정)
   - [BLOCK] 있음 → Round 2

### Round 2 ([BLOCK] 있을 시)
4. Leader가 [BLOCK] 반영하여 설계 수정
5. Codex 재검토 (동일 포맷)
6. 결과 처리:
   - [BLOCK] 해소 → 위험도 점수 확인 후 Judge 결정
   - [BLOCK] 지속 → AskUserQuestion (무한 루프 없이 중단)

## 출력 형식
```
## Debate 결과 (Round N)

채택 결정: {최종 선택}
위험도 점수: {X}/9 → Judge: Leader / 사용자

수용된 Codex 비판:
- [분류] {비판} → {어떻게 반영}

반박한 Codex 비판:
- [분류] {비판} → {반박 근거: 의견만이면 기각 가능}

최종 근거: {왜 이 결정인가}
```

## 비용 통제
- Codex xhigh: 문서 길이에 따라 다름 (토큰 절단 방지 위해 설계안 3000자 이내 권장)
- 총 라운드: 1-2회 고정
- 대상: 불확실성/영향도/복잡도 합계 6점+ 안건만

## 제약 및 안전장치
- Codex 미가용 시: 경고 출력 + 위험도 8+ 안건은 사용자에게 수동 검토 요청
- 2라운드 후 [BLOCK] 지속 시: 무조건 AskUserQuestion (자동 해결 시도 금지)
- 토큰 절단 방지: 설계안을 파일로 저장 후 -C 옵션으로 전달
- 기획 단계가 아닌 구현 중 발견되는 구조 리스크도 debate 진입 가능 (v1 제한 완화)
