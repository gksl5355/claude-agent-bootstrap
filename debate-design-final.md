# Debate Mode 최종 설계 v3
# (Codex xhigh 2라운드 debate 결과 반영)
# Round 1: [BLOCK] 4개 → 전부 수용
# Round 2: [BLOCK] 6개 → 4개 수용, 2개 부분수용(오버엔지니어링 기각)

## 목적
기획/아키텍처 결정 전, Leader의 설계안을 Codex xhigh가 적대적으로 검토하여
blind spot을 제거하고 더 나은 결정을 도출한다.

## 트리거 (하드 vs 소프트)

**하드 트리거 (무조건 debate):**
- `irreversible=true`: 되돌리기 어려운 결정 (DB 스키마, 외부 API 계약, 인증 방식)
- `영향범위=3`: 전체 시스템에 영향 (공용 타입, 공유 미들웨어, 배포 파이프라인)

**소프트 트리거 (위험도 점수 6+ 시 debate):**
- 사용자가 "debate", "아키텍처 토론" 명시 요청
- 기술 선택지 2개 이상이고 팀 전체에 영향

**소프트 트리거 제외:**
- 일반 "검토해줘", "리뷰해줘" → 별도 review 플로우 (debate X)

## 위험도 점수 (각 1-3점, 합계로 Judge 결정)

| 축 | 1점 | 2점 | 3점 |
|----|-----|-----|-----|
| 불확실성 | 검증된 패턴 | 일부 불확실 | 실험적/전례 없음 |
| 영향 범위 | 단일 서비스 내부 | 2개 도메인 이하 | 전체 시스템/공용 |
| 복잡도 | 단순 구현 | 중간 | 크로스 레이어/멀티 팀 |

- 합계 6-7점 → Leader가 Judge, 근거 문서화
- 합계 8-9점 또는 하드 트리거 → 사용자가 최종 Judge (AskUserQuestion)

**자기채점 우회 방지**: 영향범위=3 또는 irreversible=true이면 점수 무관하게 하드 트리거 적용.
불확실성 이견 시: 높은 점수 채택 (보수적 기준).

## 참여자

- **Proposer**: Leader (Sonnet+thinking) — 설계안 초안 작성
- **Critic**: Codex exec xhigh — 적대적 검토 (read-only)
- **Judge**:
  - 저위험 (6-7점): Leader, 단 BLOCK 이견 시 → 자동 사용자 에스컬레이션
  - 고위험 (8-9점 / 하드 트리거): 사용자 (AskUserQuestion)

## 설계안 입력 형식 (필수 필드)

```
## 결정 대상: {무엇을 결정하는가}
## 컨텍스트: {프로젝트 현황, 기존 아키텍처}
## 제안 방향: {채택하려는 방향과 근거}
## 고려한 대안: {검토 후 기각한 이유}

## 비기능 요구 (필수)
- 성능/처리량: {예: 응답시간 < 200ms, TPS 1000+}
- 비용 한계: {예: 월 $X 이내 / 토큰 비용 포함}
- 보안/규제: {예: GDPR, 암호화 필요 여부}
- 가용성/SLO: {예: 99.9% uptime}
- 롤백 기준: {언제 되돌릴 것인가, 방법은}

## 위험도 점수
- 불확실성: {1/2/3}  영향범위: {1/2/3}  복잡도: {1/2/3}
- 합계: {X}/9 | irreversible: {true/false}

## 우려사항 (자기검열): {스스로 의심되는 부분}
```

## Critic 출력 강제 포맷

Codex에 요청 시 반드시 아래 포맷 준수를 지시:

```
[BLOCK|TRADEOFF|ACCEPT] {분류}: {한 줄 요약}
- 문제: {구체적으로 무엇이 문제인가}
- 영향: {발생할 수 있는 결과}
- 근거: {사실/제약/원칙 — 의견만이면 TRADEOFF 이하로 분류}
- 수정안: {구체적인 대안}
- 미반영 리스크: {이 비판을 무시하면 어떤 일이 생기는가}
```

포맷 미준수 시: 1회 재질의 후 실패 처리.
증거 없이 의견만 있는 비판 → Proposer가 "근거 부족"으로 기각 가능.
단, BLOCK 기각 이견 → 자동 사용자 에스컬레이션.

## 분류 기준

- **[BLOCK]**: 핵심 요구를 충족할 수 없거나 출시/구현이 불가능한 경우
  - 예: 데이터 정합성 파괴, 보안 취약점 내재, SLO 달성 불가
- **[TRADEOFF]**: 요구는 충족하나 비용/복잡도/유지보수 부담이 명확히 증가하는 경우
  - 예: 성능 개선을 위해 복잡도 2배 증가
- **[ACCEPT]**: 즉시 반영 가능한 개선점. 미반영 시 품질 저하

## CLI 전달 방식 (토큰 절단/쉘 깨짐 방지)

인라인 문자열 대신 임시 파일로 전달:

```bash
# 설계안을 파일로 저장
cat > /tmp/debate-input.md << 'DEBATE_EOF'
{설계안 내용}
DEBATE_EOF

# Codex에 파일 기반으로 전달
codex exec -c model_reasoning_effort=xhigh -s read-only \
  "$(cat /tmp/debate-input.md)" 2>&1
```

설계안 길이가 3000자 초과 시: 핵심 섹션(결정 대상, 비기능 요구, 위험도)만 포함.

## 라운드 구조 (최대 2라운드 + 예외 1회)

### Round 1
1. Leader가 설계안 작성 (위 형식)
2. Codex에 전송 (파일 기반)
3. Leader 분석:
   - [BLOCK] 없음 → 조기 종료
   - [BLOCK] 있음 → Round 2

### Round 2 ([BLOCK] 있을 시)
4. Leader가 [BLOCK] 반영하여 설계 수정
   - 각 BLOCK에 대해 검증 계획 명시: "어떻게 이 BLOCK이 해소됐는지 확인할 것인가"
5. Codex 재검토
6. 결과 처리:
   - [BLOCK] 해소 → 위험도 점수 기준 Judge 결정
   - [BLOCK] 지속 → AskUserQuestion (아래 포맷)

### 예외 Round 3 (새 증거 제출 시만)
- 기준: Round 2에서 새로운 제약/사실이 발견되어 설계 전제가 바뀐 경우만
- 그 외 반복은 AskUserQuestion으로 에스컬레이션

## AskUserQuestion 포맷 (계약 명시)

[BLOCK] 지속 시:
```
질문: "Codex가 {X} 라운드 후에도 [BLOCK]을 해소하지 못했습니다. 어떻게 진행할까요?"
옵션:
  1. Leader 직접 결정 (리스크 감수)
  2. 설계 방향 재검토 (현재 안 보류)
  3. 추가 라운드 (새 증거 있을 때만)
무응답 시 기본값: 옵션 2 (보수적 fail-safe)
```

수용된 Codex 비판:
