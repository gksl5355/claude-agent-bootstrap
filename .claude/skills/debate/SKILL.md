# Debate Mode — 아키텍처 결정 적대적 검토

## 개요

기획/아키텍처 결정 전, 설계안을 Codex xhigh가 적대적으로 검토하여
blind spot을 제거하고 더 나은 결정을 도출한다.

**사용 시기:**
- `/debate "JWT vs Session Auth"` — 단독 호출
- spawn-team 내에서 하드/소프트 트리거 발동 시 자동 진입

**단독 모드:**
- argument가 있으면 → "결정 대상"으로 사용
- argument가 없으면 → AskUserQuestion으로 대화형 수집

---

## Step 1: 진입 판단

### 하드 트리거 (무조건 debate)
- `irreversible=true`: DB 스키마, 외부 API 계약, 인증 방식 등 되돌리기 어려운 결정
- `영향범위=3`: 전체 시스템 영향 (공용 타입, 공유 미들웨어, 배포 파이프라인)

### 소프트 트리거 (위험도 합계 6+ 시)
- 사용자 명시 요청 ("debate", "아키텍처 토론") — 일반 "검토해줘"는 해당 없음
- 기술 선택지 2개 이상 + 팀 전체 영향

### 위험도 점수 매트릭스 (각 1-3점, 합산)

| 축 | 1점 | 2점 | 3점 |
|----|-----|-----|-----|
| 불확실성 | 검증된 패턴 | 일부 불확실 | 실험적/전례 없음 |
| 영향 범위 | 단일 서비스 내부 | 2개 도메인 이하 | 전체 시스템/공용 |
| 복잡도 | 단순 구현 | 중간 | 크로스 레이어/멀티 팀 |

- 합계 6-7점 → Leader가 Judge, 근거 문서화
- 합계 8-9점 또는 하드 트리거 → 사용자가 최종 Judge (AskUserQuestion)

**자기채점 우회 방지**: 영향범위=3 또는 irreversible=true이면 점수 무관하게 하드 트리거 적용.
불확실성 이견 시: 높은 점수 채택 (보수적 기준).

---

## Step 2: 설계안 작성

단독 모드에서 argument가 있으면 "결정 대상"으로 바로 사용.
없으면 AskUserQuestion으로 아래 필드를 수집한다.

### 필수 필드

```
## 결정 대상: {무엇을 결정하는가}
## 컨텍스트: {프로젝트 현황, 기존 아키텍처}
## 제안 방향: {채택하려는 방향과 근거}
## 고려한 대안: {검토 후 기각한 이유}

## 비기능 요구 (필수)
- 성능/처리량: {예: 응답시간 < 200ms, TPS 1000+}
- 비용 한계: {예: 월 $X 이내 / 토큰 비용 포함}
- 보안/규제: {예: GDPR, 암호화 필요 여부}
- 가용성/SLO: {예: 99.9% uptime}
- 롤백 기준: {언제 되돌릴 것인가, 방법은}

## 위험도 점수
- 불확실성: {1/2/3}  영향범위: {1/2/3}  복잡도: {1/2/3}
- 합계: {X}/9 | irreversible: {true/false}

## 우려사항 (자기검열): {스스로 의심되는 부분}
```

### 토큰 효율 규칙
- **목표**: 설계안 1500자 이내. 최대 허용 3000자.
- 비기능 요구: 항목당 한 줄 (`성능: <100ms, 1000 concurrent` 형태)
- 컨텍스트: 3문장 이내 요약
- 대안: 기각 이유 한 줄씩
- 3000자 초과 시: 핵심 섹션(결정 대상, 비기능 요구, 위험도)만 포함하고 나머지 생략.

## Step 3: Codex 비판

### CLI 전달 방식 (파일 기반)

인라인 문자열 대신 임시 파일로 전달하여 토큰 절단/쉘 깨짐 방지:

```bash
# 설계안을 파일로 저장
cat > /tmp/debate-input.md << 'DEBATE_EOF'
{설계안 내용}
DEBATE_EOF

# Codex에 파일 기반으로 전달
codex exec -c model_reasoning_effort=xhigh -s read-only \
  "$(cat /tmp/debate-input.md)" 2>&1
```

### Critic 출력 강제 포맷

Codex에 요청 시 반드시 아래 포맷 준수를 지시:

```
[BLOCK|TRADEOFF|ACCEPT] {분류}: {한 줄 요약}
- 문제: {구체적으로 무엇이 문제인가}
- 영향: {발생할 수 있는 결과}
- 근거: {사실/제약/원칙 — 의견만이면 TRADEOFF 이하로 분류}
- 수정안: {구체적인 대안}
- 미반영 리스크: {이 비판을 무시하면 어떤 일이 생기는가}
```

- 포맷 미준수 시: 1회 재질의 후 실패 처리
- 증거 없이 의견만 있는 비판 → "근거 부족"으로 기각 가능
- 단, BLOCK 기각 이견 → 자동 사용자 에스컬레이션 (Leader 단독 결정 금지)

### 분류 기준

- **[BLOCK]**: 핵심 요구를 충족할 수 없거나 출시/구현이 불가능한 경우
  - 예: 데이터 정합성 파괴, 보안 취약점 내재, SLO 달성 불가
- **[TRADEOFF]**: 요구는 충족하나 비용/복잡도/유지보수 부담이 명확히 증가하는 경우
  - 예: 성능 개선을 위해 복잡도 2배 증가
- **[ACCEPT]**: 즉시 반영 가능한 개선점. 미반영 시 품질 저하

## Step 4: 라운드 처리 (최대 2라운드 + 예외 1회)

### Round 1
1. 설계안 작성 (Step 2 형식)
2. Codex에 전송 (파일 기반)
3. 결과 분석:
   - [BLOCK] 없음 → 조기 종료
   - [BLOCK] 있음 → Round 2

### Round 2 ([BLOCK] 있을 시)
4. [BLOCK] 반영하여 설계 수정
   - 각 BLOCK에 대해 검증 계획 명시: "어떻게 이 BLOCK이 해소됐는지 확인할 것인가"
5. Codex 재검토
6. 결과 처리:
   - [BLOCK] 해소 → 위험도 점수 기준 Judge 결정 (Step 5)
   - [BLOCK] 지속 → AskUserQuestion

**Round 2 범위 제한 (무한 비판 방지):**
- Round 2의 Codex 검토는 **기존 BLOCK 해소 여부만 판정**한다.
- Round 2에서 새로 발견된 이슈는 [BLOCK]이 아닌 **[TRADEOFF]로 문서화**하고 종료.
- 새 이슈를 [BLOCK]으로 올릴 수 있는 유일한 조건: 설계 전제 자체가 바뀐 경우 (예외 Round 3).

### 예외 Round 3 (새 증거 제출 시만)
- 기준: Round 2에서 새로운 제약/사실이 발견되어 설계 전제가 바뀐 경우만
- 그 외 반복은 AskUserQuestion으로 에스컬레이션

---

## Step 5: Judge 결정

- **합계 6-7점** → Leader가 Judge, 근거 문서화
- **합계 8-9점 또는 하드 트리거** → 사용자가 최종 Judge (AskUserQuestion)
- **BLOCK 기각 이견** → 점수 무관하게 사용자 에스컬레이션

### AskUserQuestion 포맷 (BLOCK 지속 시)

```
질문: "Codex가 {X} 라운드 후에도 [BLOCK]을 해소하지 못했습니다. 어떻게 진행할까요?"
옵션:
  1. Leader 직접 결정 (리스크 감수)
  2. 설계 방향 재검토 (현재 안 보류)
  3. 추가 라운드 (새 증거 있을 때만)
무응답 시 기본값: 옵션 2 (보수적 fail-safe)
```

---

## Step 6: 결과 문서화

### 출력 형식

```
## Debate 결과 (Round {N})

채택 결정: {최종 선택}
위험도: {X}/9 | irreversible: {true/false} → Judge: Leader / 사용자
라운드 수: {N}

수용된 Codex 비판:
- [{분류}] {비판} → {반영 방법} | 검증 계획: {언제 어떻게 확인}

반박한 Codex 비판:
- [{분류}] {비판} → {반박 근거}

미결 TRADEOFF (문서화):
- [{분류}] {내용} → {인지하고 감수하는 이유}

최종 근거: {왜 이 결정인가}
```

### 결과 저장

단독 모드 실행 시 결과를 임시 파일에 저장하여 세션 내 참조 가능:

```bash
cat > /tmp/debate-result-$(date +%Y%m%d-%H%M%S).md << 'RESULT_EOF'
{위 출력 형식 내용}
RESULT_EOF
```

---

## Codex 불가 시 fallback

Codex가 사용 불가능한 경우:

1. **경고 출력**: "Codex를 사용할 수 없습니다. Debate 프로토콜을 실행할 수 없습니다."
2. **소프트 트리거 (위험도 6-7)**: 경고 후 Leader가 자체 검토하여 진행
3. **하드 트리거 또는 8+ 안건**: AskUserQuestion으로 사용자 수동 검토 요청

```
질문: "Codex 없이 고위험 설계 결정을 진행해야 합니다. 어떻게 할까요?"
옵션:
  1. Leader 자체 검토 후 진행 (리스크 감수)
  2. 사용자가 직접 검토
  3. 보류 (Codex 가능할 때 재시도)
```

## 운영 규칙

- **비용 통제**: Codex xhigh 입력 3000자 이내 권장
- **라운드 상한**: 기본 2회 + 예외 1회 (최대 3회)
- **진입 기준**: 하드 트리거 또는 위험도 6+
- **무한루프 금지**: 2라운드 + 예외 1회 후에도 [BLOCK] 지속 시 반드시 AskUserQuestion으로 에스컬레이션
- **참여자**: Proposer(Leader/사용자) → Critic(Codex xhigh) → Judge(위험도 기반 결정)
