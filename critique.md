## 4. omc fork vs 플러그인

**플러그인 우선 권장**
- 업스트림 동기화 비용, 충돌 리스크 훨씬 낮음
- 현재 요구 대부분은 팀 스킬/프롬프트/정책 계층에서 해결 가능

**fork가 필요한 경우**
- 핵심 런타임 훅 변경이 있을 때만

**권장**: 플러그인으로 1차 구현 → 반드시 필요한 코어 수정만 최소 fork (하이브리드)

---

## 5. 에이전트 생명주기 리스크

- 피어 직접 통신 강화 → 상태 일관성(누가 최종 truth인지) 깨질 위험
- idle 유지 비용 가정이 맞아도 → 컨텍스트 드리프트, 워크트리 분기 충돌은 별개 문제
- **circuit breaker 없음**: 2회 실패 → debugger 호출까지만, debugger 실패/무효 조언/반복 실패 처리 없음
- **종료 조건 낙관적**: flaky test / 비결정적 테스트 미고려

---

## 6. 놓친 엣지 케이스

- 모노레포/다중 프레임워크 → `routes|pages|views` 규칙 깨짐
- 공유 타입/스키마/마이그레이션 → 도메인 경계를 본질적으로 넘는 작업
- 테스트 프레임워크 미존재/불안정/실행시간 과다 → fallback 없음
- 팀 스폰 중 부분 실패 (일부 에이전트만 생성) → 롤백 절차 없음
- Codex CLI 실패/미설치/권한 오류 → degrade 전략 없음
- 보안 리뷰를 Codex 1회로 마감 → 과신 (의존성/시크릿/라이선스 스캔 분리 필요)

## 7. 복잡도 대비 가치

현재 스코프 = "오케스트레이터 제품" 수준.
초기 가치 대비 운영 복잡도와 고장 면적이 너무 크다.

특히 `ralph + Codex 자동검증 + HUD + Notification + rate-limit daemon` 동시 도입은
초기 실패 확률을 과도하게 높인다.

**권장 3단계**
1. **MVP**: 동적 팀 구성 + 도메인 분할 + unit-tester 루프 + 최종 Codex 리뷰 1회
2. **안정화**: isolated 워크트리 / 롤백 / 실패 복구 / 라우팅 점수화
3. **확장**: ralph 자동 반복, HUD, 알림, rate-limit daemon
